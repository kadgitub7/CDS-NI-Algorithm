1. Decision Tree Structuring (Algorithm 1)
Feature exclusion at m > 2 is potentially over-aggressive. Algorithm 1 line 8 is:
O^km_fkm = O^k(m-1)_fk(m-1) - {1, ..., k}
This removes ALL features with indices 1 through k from the parent's feature set when building level m > 2 nodes. The intent is to prevent symmetric branches (e.g., sex→age and age→sex being equivalent). However, the set {1, ..., k} is being interpreted as the integer range up to the branching feature index k, not just removing the specific feature k. Depending on how features are indexed, this could prune valid, non-redundant features along with the intended ones.
The paper says only the specific previously-used branching feature should be excluded. If k in the code is a feature index (e.g., sex = index 1), then {1, ..., k} = {1}, which is just sex itself, and this is fine. But if k represents an ordinal position across a larger set, you'd be silently removing diagnostic features that should still be available deeper in the tree.
There's also no explicit criterion in Algorithm 1 for which feature k to select for branching. The paper implies it's chosen based on the conditional probability P(f^k_m), but Algorithm 1 iterates over all features and doesn't rank or select the best one. The actual tree structure (e.g., why sex is chosen at level 2) is determined implicitly by how CDS_Paper_Algorithms.py constructs the tree — a component you can verify for this selection logic.


2. Bin Probability Calculations (Algorithm 2)
Healthy range boundaries use bin edges rather than raw values. When computing the normal range, the implementation sets:
pythonb_min = disc.bin_edges[min_healthy_bin]      # left edge of lowest healthy bin
b_max = disc.bin_edges[max_healthy_bin + 1]  # right edge of highest healthy bin
The paper's Eq. 5 states: "100% of the values of features of Users without diseases fall in this range." Using raw min/max of healthy users (not bin edges) is the most literal interpretation. The bin-edge approach widens the range slightly (the left edge of the minimum healthy bin is ≤ the actual minimum healthy value). This is a deliberate engineering decision, but it has a concrete consequence: it makes the healthy ranges wider than they need to be, reducing the chance of an abnormal value triggering an alarm in Algorithm 4.
This is likely contributing to your missed diagnoses (false negatives). You can test this by switching to raw min/max and comparing results.
Related issue — P(h,f) denominator in Bayesian tables. The implementation uses:
pythonp_h_and_f[ci] = node.health_dist.get(cls, 0) / max(node.n_users, 1)
This computes P(h|f) (class h given branch f), not the true joint P(h, f) = n(class h in branch) / N_global. As noted in the code's own comment, this was a deliberate "correction to Bug #2". The consequence: Σ_h P(h,f) = 1 in this implementation, whereas the true joint would sum to < 1 (some of the global probability mass is in other branches). This affects P(B̂) (the evidence term in Eq. 4), which is then renormalized — making the posterior P(h|B̂) correct in relative terms but not absolute. The AF calculation is unaffected (the P(h,f)/P(h>1,f) ratio cancels the branch denominator), but the Bayesian tables used for any other downstream use would be subtly wrong.

3. Action Weight Assignment (Algorithm 2)
Action weights use a discretized bin approach that's inconsistent with the range check used in Algorithms 3 and 4. The weight computation:
pythonp_below = float(p_bin_h[:min_healthy_bin].sum())
p_above = float(p_bin_h[max_healthy_bin + 1:].sum())
r_o_h = p_below + p_above
This counts probability mass in bins entirely OUTSIDE the healthy bin span. But Algorithms 3 and 4 check raw values against the bin-edge boundaries with strict inequalities (value < b_min or value > b_max). A disease user whose raw value sits in the same bin as a healthy user's boundary value but still technically outside the strict raw boundary would be counted in the weight but NOT flagged in practice. This creates a systematic overestimate of action weights for features with borderline healthy/disease separation.
A specific consequence you can see in the wrongUserData: Many features have b_min = 0.0 and the user's value is also 0.0. Because of the strict inequality (0.0 < 0.0 is False), these users are never alarmed even though their value is at the exact boundary. This appears repeatedly across the trace.

4. Action Refinement (Algorithm 3)
The reset_per_h default is wrong and contradicts the paper. The function signature is:
pythondef run_algorithm3(..., reset_per_h: bool = False, ...):
The code's own docstring describes reset_per_h=False as "legacy, incorrect" — but it's the default. The main() call correctly passes reset_per_h=True, but any call to run_algorithm3 using the default (e.g., from other code or tests) silently uses the wrong pruning mode. That said, there's a deeper issue here:
The paper's pseudocode actually shows global accumulation (reset_per_h=False). The initialisation buffer = 0, newinf = ∅ appears BEFORE the outer for h = 2 to H do loop. This means the paper intends newinf and buffer to accumulate globally across all disease classes. The code's "§8.8 user override" decision to use per-h reset was made to improve practical results, but it deviates from the paper. The fact that the comment labels the paper-faithful mode as "incorrect" is a documentation red flag — it means there are two competing interpretations and the code has resolved the ambiguity by departing from the pseudocode without clear justification in the paper itself.
Algorithm 3 over-prunes minority disease classes, making them undetectable. For user 1 (class 6), the trace clearly shows h=6: buf_size=0 at the root node. Class 6 has zero retained actions, so it can never trigger an alarm. This happens when a minority class has too few training users with values distinctly outside the healthy range — Algorithm 3 prunes everything because the set-cover never increases. This is a structural limitation of the greedy set-cover approach when applied to small classes: the first feature retained by a larger class flags users that also satisfy the smaller class's newinf criterion, so when the smaller class is processed, its features appear to add nothing new. With per-h reset this is partially mitigated, but small classes can still lose all actions if their users are statistically "hidden" inside the healthy range due to the bin-edge boundary widening from issue 2.

5. AF Calculation
AF is capped at 1.0, which causes premature HEALTHY decisions. The _update_AF function clamps to [0, 1]. Once AF saturates at 1.0 (as happens at the female node for user 1 after processing h=2's actions), rw = 0.0. Since 0.0 ≤ 0.025 = threshold, the user is immediately declared HEALTHY at the end of the disease loop — regardless of what the remaining h=3...H checks find. No alarm is triggered because the remaining checks (h=3, 4, ...) still run and correctly find the user in-range, but the rw threshold was already met from h=2's accumulation alone.
The paper never states AF should be capped at 1.0. The capping is introduced as an "engineering choice" based on fuzzy-logic semantics, but it creates a dangerous interaction: high AF driven by a non-relevant disease class (h=2) pre-empts the detection of the actual disease class (h=6).
The AF accumulates per-disease rather than jointly. Eq. 7 uses r_{j|h} for the specific disease h being checked. As the CDS cycles through h=2, 3, 4, ..., each disease class's actions contribute to the same running AF. This means a user who happens to have many borderline-normal features across many disease classes will accumulate AF quickly and be declared healthy, even if one specific disease class (the correct one) was never effectively tested. There's no re-weighting or discount for the fact that passing tests for h=2 doesn't say much about h=6.

The paper doesn't specify an ordering, but processing minority classes (e.g., h=14, 15, 16) that have zero retained actions in every fold wastes no time but contributes nothing. Every LOOCV result is effectively un-reproducible with the demo path, and the two modes will produce different sets of retained actions and therefore different predictions.

B. Initial Action c0 is Double-Applied in the Main Disease Loop
The initialization block selects a random feature j_init, computes its AF contribution for h_init, and applies it. This is correct per the paper. But then in _predict_at_node, C_buf for disease h_init is loaded freshly from alg3_output.retained_for_node_disease(nid, h_init) with no exclusion for the already-applied j_init:
pythonC_buf: List[ExecutiveActionEntry] = list(sorted_actions_h)  # fresh, j_init still present
Since j_init was the highest-weight action for h_init (it was selected by max(j_actions, key=lambda a: a.action_weight)), the RL will eventually select j_init again during the main loop for h_init. The feature is measured a second time (same value) and its delta_AF is added a second time.
This is directly visible in the trace: user 1 shows QRS_angle (feat=9) at PAC=1 (initialization) AND at PAC=23 (main loop, h=2), both under the root node, same disease class. The duplicate inflates AF and wastes PAC budget. The fix is to exclude j_init from C_buf when processing h_init:
python# At the start of the h loop, when h == h_init:
if h == h_init and initial_action is not None:
    sorted_actions_h = [a for a in sorted_actions_h
                        if a.feature_idx != record.initial_action_feat]

In Algorithm2.py, run_gender_feature_analysis_once uses:

male_mask   = (gender_values == 1) & valid_gender_mask   # 1 = male

The gender analysis function therefore has males and females swapped — every permutation importance score, every healthy range table, and every "male vs female" comparison it prints is labelled backwards.

D. Per-Gender Healthy Accuracy Excludes SCREENING
In _recompute_stats, the overall healthy-correct count correctly includes both HEALTHY and SCREENING outcomes:

    if r.true_is_healthy and r.decision != HealthDecision.UNHEALTHY  # HEALTHY + SCREENING

But the per-gender breakdown uses only the HEALTHY decision:

    1 for r in self.records
    if r.true_is_healthy
    and r.decision == HealthDecision.HEALTHY   # ← missing SCREENING
    and self.data[r.user_global_idx, 1] == 0

A healthy user sent to SCREENING is counted as correct in the overall specificity but as incorrect in the per-gender breakdown. The per-gender confusion matrix therefore understates both male and female specificity, and any fairness metrics derived from it (SPD, DI, EO) are computed on wrong numbers. E. N_kf Shortcut in compute_healthy_range is Tied to the Bin-Edge Approach
The current N_kf calculation uses an integer shortcut:
pythonn_kf = float(max_healthy_bin - min_healthy_bin + 1)
This is algebraically equivalent to (b_max - b_min) / delta_b only because b_min and b_max are bin edges (evenly spaced by delta_b). If you apply Fix 2A from the earlier analysis (switching to raw min/max), b_min = healthy_vals.min() and b_max = healthy_vals.max() will not generally be bin edges, and the shortcut will produce a wrong result. The two changes must be made together:
python# After Fix 2A:
b_min = float(healthy_vals.min())
b_max = float(healthy_vals.max())
n_kf  = (b_max - b_min) / disc.delta_b if disc.delta_b > 0 else 1.0
This is not currently broken, but it is a latent bug that will be activated by the other fix.

F. Initial Action Selects a Random Feature Then Maximises Over Disease Classes, Not a Truly Random Action
The paper says c0 should be "randomly selected from C^{Om=1}_{m=1|h>1}(FA=0)" — an action is a (feature, disease_class) pair, so a uniform random draw should pick from all pairs. The implementation instead:

Picks a random feature j_init from the set of all root feature indices
For that feature, picks the disease class h_init with the MAXIMUM action weight

pythonroot_feats = sorted({a.feature_idx for a in candidates})
j_init_candidate = random.choice(root_feats)
...
initial_action = max(j_actions, key=lambda a: a.action_weight)
This biases the initial action toward features that are paired with a high-weight disease — it is not a uniform random draw over all (feature, disease) pairs. More practically, since h_init is the maximum-weight disease for j_init, and the RL in the main loop for h_init also selects maximum weight first, j_init is very likely to be selected first in C_buf anyway. The initialization therefore effectively deterministically pre-selects the best action for h_init under the guise of a random draw, then re-applies it in the main loop (bug B above).
The paper-faithful fix is:
pythoninitial_action = random.choice([a for a in root_all_actions
                                 if not np.isnan(_get_raw_value(user_global_idx,
                                                                 a.feature_idx, data))])
j_init = initial_action.feature_idx
h_init = initial_action.disease_class

G. Algorithm 3's Vectorised Inner Loop Replaces a Per-User Loop That Is Now Dead Code Inside a Triple-Quote Block
In _refine_one_node around line 750, the per-user loop is wrapped in a '''...''' string literal (making it dead code), and the vectorised replacement runs immediately after:
python'''
raw_val = _get_raw_value(global_u, o, data)
if _is_outside_range(raw_val, b_min, b_max):
    ...
    newinf[local_i] = True
'''

The vectorised code is functionally equivalent and correct. But the > and < in the vectorised form are applied to the FULL raw array including NaN values. The NaN exclusion is handled by valid_mask = ~np.isnan(raw_vals) and flag_mask = valid_mask & outside_mask, which correctly prevents NaN values from triggering flags. However, the comparison raw_vals > b_max when raw_vals contains NaN produces False in NumPy (NaN comparisons are always False), so valid_mask is technically redundant for the outside-range check — but not harmful.
The issue is purely structural: the dead code block makes the file harder to read and understand, and any future maintainer may be confused about which version is active. The dead block should be removed entirely.

H. run_loocv Passes train_data to Algorithm 3 but Algorithm 3's analyse_coverage Uses a Separate data Argument Inconsistently
The run_loocv function passes training-set data consistently:

However, the analyse_coverage utility in Algorithm 3 is called in main() with the full dataset:

analyse_coverage does:

where h_users = [u for u in node.user_indices if labels[u] == h]. The node.user_indices here are training-set indices (from tree_i), but data is the full dataset. Calling data[train_set_index] on the full array gives the wrong user's features whenever the held-out user falls at or before that index.

2. NaN Branching Feature Produces Incorrect SCREENING Decision
When a test user has NaN for the sex feature (the level-2 branching feature), _find_applicable_node at focus level 2 finds no matching node and returns None. The outer loop in run_algorithm4 then breaks out of the focus loop with decision still equal to HealthDecision.UNKNOWN from the level-1 call (which returned UNKNOWN to signal focus escalation should occur). The finalisation block then converts UNKNOWN → SCREENING.
The correct behaviour for this user, per the paper, is to use the level-1 decision. If level-1's rw > threshold, the user should have been sent to SCREENING by level-1 itself. But if level-1 returned UNKNOWN specifically because it wanted to escalate, and the escalation fails due to NaN, the user lands in SCREENING through the wrong code path. The correct fix:
pythonif active_node is None:
    log.debug(f"  m={current_focus}: no applicable node → keeping previous decision")
    # Don't escalate further; the previous focus level's decision was UNKNOWN
    # but we can't escalate, so treat as SCREENING per paper line 41-42
    decision = HealthDecision.SCREENING
    break
This is the same outcome but reached explicitly through the policy path, not as an unhandled fallback.

3. run_test_cases Uses Models Trained on Full Dataset Including Test Users
run_test_cases in main() calls build_decision_tree(data, labels) and trains Algorithms 2 and 3 on all 452 users, then runs Algorithm 4 on specific users from that same dataset. Every test user was part of training, so the healthy ranges are defined partly from their own values. For a healthy test user, this guarantees their values are within the healthy range by construction, so they can never trigger a false alarm. For a diseased test user, their out-of-range values contributed to Algorithm 3's newinf, so the action weight biases are derived partly from their own data.
The results from run_test_cases are therefore systematically optimistic and not comparable to the LOOCV results. Any accuracy figures quoted from test cases are not the paper's LOOCV accuracy. There is no in-code warning about this data leakage. The fix is either to use a held-out fold for each test case, or clearly label the test case output as "in-sample sanity check (not LOOCV)".

5. Algorithm 4's Disease-Class Loop at Level 2 Repeats ALL of h=2...H, Including Classes Already Fully Processed at Level 1
When focus escalates from level 1 to level 2, _predict_at_node at the new node processes node_disease_classes which is derived from active_node.health_dist — all disease classes present in the female (or male) training branch. This re-runs h=2, h=3, etc. from scratch at the new node with fresh C_buf.
The paper says (Section VI.B step 2): "Check if the User has the disease hdh ∈ {hd2,...,hdH}". The high-level description implies the loop restarts at hdh=hd2 at the new focus level. The implementation correctly matches this. However, the combined AF carries over. This means the AF accumulated from h=2's 42 actions at level 1 is still in AF_real when level 2 begins processing h=2 again. The level-2 h=2 actions add more AF on top of an already-substantial level-1 h=2 contribution. This makes AF saturate faster at level 2 than it would if AF were reset, contributing to the premature HEALTHY decisions seen in the wrongUserData.
This is not a deviation from the implementation's documented design ([PAPER §8.11 user override; CORRECTION TO BUG #9]), but it is an undocumented deviation from what the paper actually says. The paper's Figure 11 shows AF starting near 0 at PAC 1 of level 2 for the average user, which is more consistent with a per-level reset than with carry-over.

6. total_pac_count in PredictionRecord Counts the Initialization PAC Separately But pac_counter Inside _predict_at_node Starts at 0, Causing a One-Off in the PAC Numbering Inside Each Node
The initialization adds record.total_pac_count += 1 directly before the focus loop. Then pac_counter = [0] resets inside the loop. Inside _predict_at_node, the first pac_counter[0] += 1 produces PAC number 1 again. This means both the initialization action and the first main-loop action are recorded with pac_number = 1 in the AF trace, visible in the wrongUserData trace (two entries with PAC=1 at the root).
For the PAC count totals this is harmless (the init count and main-loop counts are summed correctly). But for the trace's pac_number field, having two different actions with pac_number = 1 at the same node makes the trace ambiguous. Any downstream analysis that uses pac_number to order events will see an ordering collision. The fix is to initialize pac_counter to [1] when entering _predict_at_node if the init action already occurred, or to treat the init PAC as PAC 0 rather than PAC 1.

7. Algorithm 3's _compute_node_summary Uses Only the Last Log Record to Determine final_buffer and n_users_flagged, Which Is Wrong After Per-h Reset
pythonnode_log = [r for r in log_records if r.node_id == nid]

With reset_per_h=True, buffer and newinf reset for each h. The last log record has the buffer value from the LAST disease class processed, not the global maximum. If disease class h=16 was processed last and had 0 actions (its C_buf is empty and h_actions_sorted is empty), the last log record may be from h=15 or even h=2, whichever was processed last.
